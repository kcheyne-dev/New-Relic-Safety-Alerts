New Relic Safety Alerts — Project Overview

Architecture · Roadmap · Operating model

This document explains what the Safety Alerts project is, how it works technically, and where it's going. It's intended for engineering leads, security/compliance reviewers, and exec sponsors who need context before approving next-step integrations.

1. What this is

A real-time threat-monitoring and crisis-communications dashboard for New Relic's Crisis Management Team (CMT). It pulls live alert feeds from public data sources, plots them against the 9 New Relic offices, lets the CMT scope incidents to specific geographic areas, dispatch coordinated safety messages, and track per-employee response status — all from a single interface.

The project has two parts:

  • Frontend — a single-file static HTML dashboard. Hosted on GitHub Pages today; can run anywhere static files run. No build step. ~3,600 lines of vanilla JavaScript + custom CSS using Leaflet for mapping.
  • Backend — a TypeScript Node service (Fastify + Postgres + PostGIS) that ingests from 12 public data sources, deduplicates across them, geocodes, matches to offices, and serves a REST + Server-Sent Events API. ~3,200 lines of TS across 44 files.
Live URL: https://kcheyne-dev.github.io/New-Relic-Safety-Alerts/
The frontend is fully usable today against mock data. The backend is built and tested but not yet wired to the frontend (deferred until vendor integrations are approved).

Frontend stack

  • Pure HTML/CSS/JS — no framework, no build step, no transpiler.
  • Leaflet + Leaflet.draw + Leaflet.markercluster (via CDN) for mapping.
  • CartoDB dark/light tiles. RainViewer for live precipitation. NASA GIBS for live land-surface temperature.
  • State management: a single STATE object + render functions that read it.
  • Persistence: localStorage with debounced saves, schema-versioned for future migrations.
  • Accessibility: ARIA labels, keyboard navigation on rails, semantic dialog roles on modals.
  • Security: CSP meta tag, escape-everything XSS prevention, HTTPS-only sources.

Backend stack

  • Node.js 20+ with TypeScript and ESM modules.
  • Fastify for HTTP — chosen over Express for speed and better TS support.
  • PostgreSQL 16 + PostGIS for spatial queries (ST_DWithin for office proximity, indexed geometry).
  • node-cron-style intervals for ingestion scheduling.
  • bcrypt + jsonwebtoken for auth.
  • pino for structured logging, zod for input validation.
  • Server-Sent Events for live push to the dashboard (no polling).

v2.x

Status strip, persistence layer, drag-resize panels, smart alert ordering, unified Map Tools panel, multi-message incident flow, reopen-incident, comprehensive incident report export, attachments + URL linkification, hazard map layers, NASA GIBS / RainViewer live tiles, CSP, accessibility pass, audit fixes.

B-1

Backend Sprint 1 — Foundation

TypeScript + Fastify scaffold. USGS + NWS adapters. Postgres schema with PostGIS. Office proximity matching. REST API for events + sources/health. In-process scheduler.

B-2

Backend Sprint 2 — Breadth + geocoding

Added EONET, GDACS, EMSC, MeteoAlarm, State Dept adapters (5 more sources). Cached Nominatim geocoder with 1 req/sec throttle. Pre-pass geocoding integrated with the persist pipeline.

B-3

Backend Sprint 3 — Dedup, SSE, city feeds

Cross-source clustering (USGS + EMSC + GDACS publishing the same quake → one event). Centralized severity normalization. Server-Sent Events stream. 4 city open-data adapters: SF Police, Atlanta APD, Portland FlashAlert, London TfL.

B-4

Backend Sprint 4 — Production hardening

GDELT firehose with theme + tone filtering. Stale-event sweeper. Source health monitor with auto-detected webhooks (Slack / PagerDuty / generic). Optional self-hosted Nominatim Docker profile.

B-5

Backend Sprint 5 — Real multi-user

Server-side incidents/messages/responses/notes/log replacing localStorage. JWT auth with bcrypt. Role-based access control (admin / cmt / office / employee). Append-only audit log for compliance. Okta migration documented as single-function swap.

Total operational cost at current scope: ~$10-30/month. The expensive parts are the (future) paid integrations — Slack/Workday/Navan/Okta licenses are typically already paid for at the org level.

Reliability

  • Stale-event sweeper auto-marks events as inactive after 24h.
  • Source-health monitor fires webhook alerts when a source has been silent >30 min — auto-detects Slack / PagerDuty / generic from URL.
  • Cross-source dedup means losing one source doesn't lose visibility on an event (USGS + EMSC + GDACS all cover earthquakes).
  • Source-row health stored in Postgres persists across restarts.

Compliance

  • Append-only audit_log table captures every authenticated mutation: user, action, target, IP, user agent, payload.
  • Append-only raw_events table preserves every payload from every source — full provenance chain from raw → normalized → clustered → incident.
  • Schema-versioned persistence in both backend (SQL migrations) and frontend (localStorage schema field) so future shape changes can be migrated cleanly.
  • RBAC built in from day one with four tiers; gating is one line per route.

9. Summary

The Safety Alerts project today is a complete operational dashboard backed by an ingestion engine that pulls from 12 free real-time data sources, with role-based auth, audit logging, and clean integration points for Slack / Gmail / Workday / Navan / Okta when those approvals come through.

What's deployed and live: the frontend dashboard at the Pages URL, using mock data that mirrors the real schema.

What's built but not deployed: the full backend — runnable locally with docker compose up -d && npm run dev, ingesting from all 12 sources.

What's gated on approvals: wiring the dashboard to the backend, vendor integrations (Slack / Gmail / Workday / Navan), Okta SSO.

What's recommended to start now: OSAC application (free, ~3 weeks of vetting, no other dependencies).
